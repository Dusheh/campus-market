## 目录分层

### Handler 层
处理 HTTP 请求，参数校验，调用 Service 层，返回响应。不包含业务逻辑。

### Middleware 层
JWT 认证、CORS、日志、限流等横切关注点。

### Service 层（待实现）
核心业务逻辑：用户认证、服务管理、订单处理、消息掠老等。

### Repository 层
数据库操作封装，提供 CRUD 接口。

### Model 层
数据模型定义，GORM 模型映射。

## 数据流

### 卖服务流程
```
接单者发布服务 → POST /api/services → Handler 校验 → Service 处理 → Repository 写入 MySQL
客户浏览服务 → GET /api/services → Handler 分页查询 → Repository 查询 MySQL
客户购买 → POST /api/orders → 创建订单 → 更新服务状态
```

### 买需求流程
```
客户发布需求 → POST /api/demands → 创建需求记录
接单者浏览需求 → GET /api/demands → 需求列表
接单者接单 → POST /api/orders → 创建订单 → 更新需求 offer_count
```

## 技术选型理由

| 选择 | 理由 |
|------|------|
| Go + Gin | 高性能、低内存、并发优势，适合校园高并发场景 |
| GORM | Go 生态最成熟的 ORM，自动迁移、预加载 |
| MySQL | 关系型数据，事务支持，适合订单/交易场景 |
| Redis | 缓存热点数据、Token 黑名单、消息队列 |
| JWT | 无状态认证，适合小程序 API 场景 |
| Docker | 环境一致性，简化部署 |
